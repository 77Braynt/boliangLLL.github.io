Vue.js个人经验总结

2018-12-16 项目总结 Comments

其实Vue里，在我们开发中，大多数的东西我们都可以在vue.js的官网里找到。但是也有些 在你封指令或者组件时候。有更便捷的方法，或者可以用高级用法来形容。

一些有关Vue的问题

Template渲染不要附带大量逻辑，逻辑放在js处理，template只负责渲染数据。
template中绑定的变量建议不要使用函数。

data中初始化的变量类型必须跟该变量类型符合，例如数组初始化为空数组，字符串初始化为‘’，数字初始化为0或者根据情况显示为空字符串，不要数组初始化为空字符串，避免compute属性或者渲染时出错。 在我们初始化的时候，一定要知道其是什么数据类型，以免渲染错误

如果data中的对象的属性为watch属性或者页面渲染属性，最好预先初始化属性的值，而不是初始化一个空对象，一作文档作用，二做初始化的getter和setter使用。详情可查看vuejs文档解释

不要在computed计算属性函数里给另一个变量赋值，或者执行请求，而是用watch实现联动。
–computed主要是对于变量值进行计算，不利于进行其它操作。
–watch 当变量发生变化的时候，进行其它操作。

Vue一些细节 - 组件化

组件设计时，应该保证组件的简单，配置项过多时，通过opts选项传入，并给予默认值，或者拆分组件。
组件传入的属性命名，应该尽量与现有html组件习惯保持一致，例如input的placeholder，disabled，maxlength，readonly等属性，select的options配置。

可以根据条件拆分组件为动态组件，而不是template直接判断条件渲染。
组件间的共同逻辑，可以使用mixins进行合并（最常用，最实在的一件组件化方法，高内聚低耦合）。

过滤器的参数，不建议设计成用1、2、3传参来代表做不同的过滤，如formatMoney，rateFormat，（虽然是自己封装的，但是还是没多懂意思，也不方便维护，其它人也难看明白。 所以后面的变量命名也规范了很多）
也不建议编写太多个传入参数，如_toFixed，建议做成配置参数+默认值，或者传入格式化参数。formatAllDate，映射尽量后端返回，因为前端不知道什么时候会增加一个枚举值。

有关安全性的问题

永远不要相信用户的输入！！！
能够在后端获取的，就不要前端传值，前端只传用户的自身输入。(例如之前的传短信验证码，前端如果传，就跳转一个成功结果页面，不传就跳转另外一个结果页面。 那么前端可以控制结果，这个不安全，如果信息给截取的话，会有问题)
关键验证不要相信前端传值区分。
前端做合法性验证，是为了用户体验，后端合法性验证才是屏障。
数据防篡改。–方法：签名算法。（大佬上次说过，半懂。下面大佬教学）
数据原子性，一致性，事务错误回滚，事务锁。
其他安全问题知识，可参考《白帽子讲Web安全》，跨站脚本攻击，跨站点请求伪造，注入攻击等。

代码规范问题

英文单词拼写，杜绝错误，更杜绝错误后不做修改。
不要拿中文来做对比，枚举值，要用变量值或者枚举值。
数字0、null，undefined，空字符串，false,NaN都是false值，其他（包括字符串0、null字符串）为true，所以一般判断是否有值或者数组是否有数据，可以使用if(a)和if(arr.length)判断即可。（arr.length最好的，表示已存在，已经是大于0的）
能够用尽量少的代码实现同样的功能，ifelse的条件合并，三元表达式。简洁的代码，可以表达更新清晰，更加容易维护，出问题概率也减少。
合理使用正则表达式做字符串替换和数据合法性判断，数字精度问题。
页面返回问题，1. 去掉返回按钮，禁止右滑，给用户选择新入口；2. 页面固定传值，例如xxx_RETURN_URL=/xxx/xxx

对于for,for in for of forEach

在项目开发中，很多人喜欢用for in 循环。 但是在这里我是不支持的。 因为当你那个数组
方法 是否可以中断 是否可以获取index 是否支持异步等待 是否直接取到item 兼容性
传统for Y Y Y N 支持
for in Y Y Y N 支持
forEach N Y N Y ES5支持
for of Y N Y Y ES6支持

for in 最初设计是以任意顺序遍历一个对象的可枚举属性，所以遍历数组时，获取到的index是字符串，还会获取到构造函数原型中继承的属性，不适合用于数组遍历。(例如你在forIn之前， 你定义一个属性，array.prototype.getFun = function()。后面也会返回这个数据)
所以，建议大部分情况，遍历数组操作，用for of。

Promise，Await，Async的一些问题

什么时候应该加async，当函数里有await等待promise完成时，该函数就必须标识为async异步函数。
不并发的异步函数，用await即可，循环可用for of。并发的异步函数，用Promise.all，循环可用map函数把数据映射成promise函数。
await的promise出现错误被catch后，后面的代码会被继续执行！所以建议在整个流程函数外统一catch错误，要么就添加返回结果的判断。

循环异步函数，出现错误时，如果希望不影响其他执行，可以统一resolve(这种情况不一定要用reject，因为不想影响其它执行)，但用变量代表状态status来区分成功与否（另外一种解决方法），而不是直接让promise reject。(例如懒加载每一张图片，不可能因为一张图片加载失败而所有图片不进行显示)

既然是异步，就不保证操作耗时快慢，所以，再快的异步操作，也要把下一步操作写在异步回调结果里！不要进入页面后setTimeout(()=>{…},1000)一秒后执行某些代码，页面间传值也是，必须在session保存完成后再跳转页面。
可以用await模拟同步代码，躲避回调地狱，但必须清楚你的代码是怎样执行的。

注意
// 异步函数可以捕获异常，但捕获处理后依然会往下执行，因为await拿到的是catch完后的返回值

// 异步函数处理完后，可以继续处理，await拿到的是最后处理的结果
// api串行请求，任意一个报错即不会继续往下执行，跳到catch捕获处理

this.$forUpdate()

this.$forceUpdate() 处理页面刷新问题（v-for循环值刷新等）
问题描述：在使用Vue框架开发时，在函数中改变了页面中的某个值，在函数中查看是修改成功了，但在页面中没有及时刷新改变后的值
解决：运用 this.$forceUpdate() 强制刷新（因为数据层次太多，render函数没有自动更新，需手动强制刷新。）
用法: 在for循环后添加 。或者点击事件的时候， 需要改变某些状态值
this.$forceUpdate(); //强制刷新，解决页面不会重新渲染的问题

因为能修改state的值。 唯一路径 还是mutations

iconfont的基本用法

1.把需要的icon添加到购物车
2.如果需要哪个的话，添加到自己的项目之中。
3.然后点击我的项目。 可以看到自己需要的图标。 和对应的图标代码。
4.然后可以在该页面点击下载至我的本地。 然后进行相关操作
5.下载iconfont.在asserts文件夹中。然后放入文件iconfont.css 创建iconfont. 放入文件iconfont.eot，iconfont.svg，iconfont.ttf，iconfont.woff。
6.对于每次下载之后。 在文件夹iconfont里面的4个文件。 需要重新替换。
7.在iconfont.css 需要更新64位的url. 把下载回来的iconfont中的64位替换之前的64位代码。– url(‘data:application/x-font-woff;charset=utf-8;base64,**–(因为第一次上面的src的路径，我们已经把绝对路径写好了。如果我们直接用最新的iconfont.css的话，又要写多一次路径。 所以直接改一次64位的就好了。 其它的就不用理了)
最后 在Main.js import ‘styles/iconfont.css’。

css3 渐变属性 background-image: linear-gradient(top, rgba(0, 0, 0, 0), rgba(0, 0, 0, 0.8));

一个div 从一个方向到另外一个方向的样式渐变。 用 linear-gradient。
第一个属性为方向（会往反方向渐变）。 第二个是初始颜色值。 第三个是到最终方向的颜色值。 —- 这是基础方法。 其中过渡的话。 可以看网站教学

用keep-alive 包住的组件里面有better-scroll— PC可以滑动。手机端偶现不能滑动

em….最近在做一个项目。 突然发现用手机测试我做的项目的时候，被keep-alive包住的Better-scroll 不能滑动。 在网上找了很多，却不能找到解决方法。 后面记起了自己原来用keep-alive包住了这个组件。… 网上的人说keep-alive会有缓存作用，也会记录组件的位置。 就算我们跳转的时候，路由设置初始化高度到顶部，也会让这个better-scroll不能滑动。
后来发现看过一些文章。 用手机测试的时候，会有一些缓存机制。 会记录上个页面的信息。 .. 会导致better-scroll不能滑动（我也没多清楚）
解决方法：就是在activated函数 重新启动better-scroll
